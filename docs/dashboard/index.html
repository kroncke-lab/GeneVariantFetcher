Active operating loops

  • Cold-start loop — `gvf gvf-run <GENE>` runs doctor, discovery/filtering, extraction, source QC/recovery, DB-observed recovery layers, corpus sync, and run reporting with source recovery enabled by default.
  • Recall-refresh loop — `scripts/refresh_recall.py` fetches new supplements, bridges corpus to harvest layout, folds source, acceptance-gates targeted re-extraction, scores, and optionally surgically lands into canonical DBs.
  • Full-coverage loop — `gvf-run --full-coverage` adds walk-to-taper priority extraction, carrier-count guard, and variantFeatures enrichment after the bounded extraction path; it remains opt-in and should be scored before promotion.
  • Dashboard loop — The overnight automation scores only when inputs changed, updates pipeline.json and strategy.json, rebuilds docs/dashboard/index.html, and publishes only dashboard-scope files from a clean worktree.

Guardrails

  • No gold leakage in default recovery — ClinVar, PubTator, source recovery, and cold-start paths default to DB-observed PMIDs; gold-assisted worklists remain diagnostic.
  • Source never-downgrade behavior — Corpus reuse and prior-run consolidation copy only usable full text; compromised stubs fall through to fresh acquisition attempts when credentials improve.
  • Extraction source fingerprints — Per-PMID JSON is reused only when source SHA/size still match, preventing stale abstract-only or stale full-text extractions from silently winning.
  • Count hygiene is opt-in by policy — COUNT_GUARD_POLICY and COUNT_CLASSIFIER_POLICY default to off; flag/clear behavior must be explicitly selected and then scored for recall, counted precision, and MAE.
  • Surgical canonical landing — `refresh_recall.py --land` backs up canonical DBs, preserves recovery-layer rows, and reverts on unique-variant, row-recall, or rows-mode MAE regression.
  • Generated metrics stay out of git — recall_metrics, SQLite DBs, local results, credentials, and scratch work stay local; only dashboard HTML/sidecars and the dashboard builder are eligible for nightly publication.

Current architecture bottlenecks

  • Metric bottleneck: counted extras and count semantics — The latest disagreement report is led by pipeline_only_extras and count_semantics, so precision/count attribution must be separated from source-acquisition work.
  • Architecture bottleneck: source acquisition and folding — Missing or abstract-only source remains large enough to block the 90% target; the architecture has folding/recovery hooks, but high-yield supplements still need bounded, authorized acquisition and refresh.
  • Architecture bottleneck: available-source underextraction — The extractor/table-router path still needs targeted exhaustion checks on PMIDs with usable source and many missing rows, rather than broad unbounded extraction reruns.
  • Architecture bottleneck: matcher lanes — Already-extracted indels, compound rows, splice/IVS records, and structural/CNV events still need explicit matcher lanes or denominator treatment.
  • Dashboard bottleneck: accepted baseline drift — The fresh score matches the July 7 nightly artifact but remains below docs/RECALL_STATUS.md by 2 unique variants, 1 row, and 1 PMID, so status updates should reconcile whether this is intended cleanup or regression.
